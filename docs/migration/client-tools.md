## 1. Purpose

These five C++ binaries are the **build-time pipeline that turns a retail/Classic WoW client install into the on-disk data trees a private worldserver loads at runtime**. They read the client's CASC storage (TACT product `wow_classic` for Wrath Classic 3.4.3.x) and emit three sibling directories under `DataDir/`:

- `maps/<MapId4>_<gx2>_<gy2>.map` — per-tile heightmap + area + liquid + holes, written by `map_extractor`.
- `vmaps/<MapId4>.vmtree` + `vmaps/<MapId4>_<gy2>_<gx2>.vmtile` — BSP-compressed WMO/M2 collision geometry, produced in two stages: `vmap4_extractor` writes raw per-model `.bin`/`dir` files into `Buildings/`, then `vmap4_assembler` converts that intermediate into the final `vmtree`/`vmtile` pair the runtime mmaps.
- `mmaps/<MapId4>.mmap` (header) + `mmaps/<MapId4><gx2><gy2>.mmtile` — Recast/Detour navmesh tiles, generated by `mmaps_generator` from the already-extracted `maps/` + `vmaps/`.

**Without these directories the worldserver is non-functional past the login screen**: terrain Z-queries fail (no `GridMap`), every line-of-sight test returns "no data" (no `VMapManager2`), and AI pathfinding short-circuits to straight-line because Detour can't load any tile. Players spawn in the air, mobs walk through walls, and combat ranged checks misbehave silently.

`extractor_common/` is the shared library all of these link against — most importantly `CASC::Storage` / `CASC::File` (a thin C++ wrapper over CascLib) and the `*LoadInfo` `DB2Meta` declarations every tool uses to read DB2 records out of the CASC storage.

## 10. Regression tests to write

These tests apply **whether we port the extractors or rely on TC's**, because the runtime side will load the extracted output regardless.

- [ ] **Header-magic round-trip**: read a TC-extracted `0001_31_31.map`, assert `mapMagic == "MAPS"`, `areaMagic == "AREA"`, `heightMagic == "MHGT"`, `liquidMagic == "MLIQ"` (exact 4-byte literals from `MapDefines.cpp`). Catches an accidental endian-flip or struct-layout drift.
- [ ] **`map_fileheader` size invariant**: `sizeof(map_fileheader) == 11×4 = 44 bytes` (4-byte magic + 10 u32). Mirror the C++ static_assert at the Rust struct level.
- [ ] **`MmapTileHeader` size invariant**: `== 20 bytes` (`static_assert` line 41 of `MMapDefines.h`); `MMAP_MAGIC == 0x4d4d4150` (`'MMAP'` little-endian); `MMAP_VERSION == 15`.
- [ ] **`VMAP_MAGIC` strings**: `b"VMAP_4.B"` (runtime) and `b"VMAP04B\0"` (raw, 8 bytes incl. null) — exact byte arrays.
- [ ] **Liquid type flag layout**: `map_liquidHeaderTypeFlags { Water=0x01, Ocean=0x02, Magma=0x04, Slime=0x08, DarkWater=0x10 }` — these go into the `.map` file and out the runtime's `LiquidData::type_flags`; an off-by-one here silently breaks "swimming" detection.
- [ ] **Tile-name format**: `format!("{:04}_{:02}_{:02}.map", mapId, gy, gx)` for maps but `format!("{:04}{:02}{:02}.mmtile", mapId, gx, gy)` for mmaps (no underscores) and `format!("{:04}_{:02}_{:02}.vmtile", mapId, gy, gx)` for vmaps with `(gy, gx)` swapped — the C++ comment at `MapTree.cpp:94-95` calls this out explicitly. A naming-convention test prevents the most common "off-by-axis" data corruption.
- [ ] **DataDir present-or-fail**: launching `world-server` against a `DataDir` that lacks `maps/`, `vmaps/`, `mmaps/` should produce a precise error pointing at this doc, not a panic deep in `Wdc4Reader`.

## 11. Notes / gotchas

- **The runtime in RustyCore *already* expects extracted data.** `wow-data/src/item.rs:43` reads `{data_dir}/dbc/{locale}/Item.db2`; the test on line 101 hard-codes `/home/server/woltk-server-core/Data`. So the data trees produced by these tools are a load-bearing input *today*, even though the tools themselves aren't ported. Any developer setting up a fresh box will hit this within minutes.

- **CASC, not MPQ.** Wrath Classic 3.4.3.54261 ships on the modern CASC/TACT pipeline, same as retail. The TC extractor's `RetardCheck()` (vmapexport.cpp:411) explicitly bails out if it sees `.MPQ` files, with a now-stale "use 3.3.5 branch" message that *predates* the WoLK Classic release. The 3.4.3 branch of TC handles wow_classic CASC fine; that comment should not be taken to mean "tools are MPQ-only". Porting to a Rust MPQ reader (the original 3.3.5-vanilla way) **would be wrong** — the 3.4.3 client has no MPQs to read.

- **CascLib licence is MIT** — no licence blocker for shipping a Rust FFI binding (`crates/wow-casc/`) or a pure-Rust port. CascLib upstream: <https://github.com/ladislav-zezula/CascLib>.

- **`wow_classic` product code** — when calling `CascStorage::Open`, pass `"wow_classic"` (not `"wow"` which is current retail; not `"wow_classic_era"` which is Vanilla Classic 1.15.x). Wrong product ⇒ tool extracts the wrong client's maps and the runtime won't load them.

- **Tile coordinate axis swap.** `.vmtile` filenames use `(MapId, gy, gx)` order; `.mmtile` uses `(MapId, gx, gy)` glued without separators. This is the single most error-prone thing in the entire pipeline — any port will get it wrong on the first attempt. The `MapTree.cpp:94-95` C++ comment is the canonical reference: `tilefilename << std::setw(2) << tileY << '_' << std::setw(2) << tileX << ".vmtile"`.

- **`FILE_FORMAT_VERSION = 18`** for the ADT/WDT chunked-file reader (`loadlib/loadlib.h:27`) — bumped from 16 in the WoLK Classic transition. Any pure-Rust `ChunkedFile` port must handle exactly v18; older `.adt` from 3.3.5 client will trip it.

- **`MmapTileHeader::usesLiquids = true`** for the WoLK config. The flag lets a tile opt out of liquid mesh, but TC always builds with it on. Rust port: keep the bool, default true, do not invent new flags.

- **Padding in `MmapTileHeader`**: the `static_assert` at `MMapDefines.h:42` is intentional — three bytes of padding are explicitly zeroed because `mmaps_generator` writes the struct via `fwrite` and any uninitialized stack bytes leak into the on-disk file. A Rust port using `bytemuck::Pod` + `repr(C, packed)` gets this for free, but **only if the struct is `Default`-zeroed before write**.

- **Off-mesh-connections file (`offmesh.txt`)** is hand-curated by TC contributors — adds jump-down / jump-across nav links Recast can't infer. Pure ASCII, dozens of lines. If RustyCore stays on TC-extracted mmaps, the file is already baked in. If we re-extract, source it from `/home/server/woltk-trinity-legacy/contrib/extractor/offmesh.txt` (location varies by branch).

- **Threading model.** `MapBuilder` runs N `TileBuilder` worker threads (default = `hardware_concurrency`), all pulling from a `ProducerConsumerQueue<TileInfo>`. Each thread holds its own `TerrainBuilder`, `rcContext`, intermediate buffers. Rust port should mirror with `rayon` or a dedicated `tokio::task::spawn_blocking` pool — *not* async.

- **Memory pressure.** Building all continents (Eastern Kingdoms, Kalimdor, Northrend) at once peaks around **8 GB RAM** with 8 threads, takes **30–90 minutes** on modern hardware. This is operator-facing, not server-facing — but it's a real constraint for CI.

- **Battlegrounds, dev maps, transports skipped by default**: `MapBuilder::shouldSkipMap` filters via `isBattlegroundMap`/`isDevMap`/`isTransportMap`. Operators with `--skipBattlegrounds=false` get those tiles too. Rust port must keep these flags or break BG match-making.

There is no "tool format" vs "runtime format" split: both sides share `MapDefines.h`, `VMapDefinitions.h`, `MMapDefines.h`. As long as RustyCore's eventual `GridMap`/`VMapManager`/`MMapManager` Rust ports reproduce these exact magic bytes, version numbers, and struct layouts (verified per #CTOOLS.3 and per the regression tests in section 10), TC binaries' output drops in unmodified.

**Practical implication: Rust port is deferred — not blocked.** Operators today should:
1. Build the TC tools from `/home/server/woltk-trinity-legacy/src/tools/` (they're already in the legacy submodule).
2. Run them against a 3.4.3 client (TACT product `wow_classic`).
3. Copy `dbc/`, `maps/`, `vmaps/`, `mmaps/` into RustyCore's `DataDir`.

The Rust port (#CTOOLS.4 through #CTOOLS.7) becomes mandatory only at the next major client version bump — i.e. when 3.4.4 / 3.4.5 ships and the TC 3.4.3 branch stops tracking. At that point the C++ tools will need re-validation against the new client and a Rust port stops being "deferred" and starts being "forced". For now: **document the workflow** (#CTOOLS.1), **fail-fast on missing data dirs** (#CTOOLS.2), **lock the wire format** (#CTOOLS.3), and leave the porting tickets dormant.

**One caveat to the binary-compat verdict**: the `vmap4_extractor`'s opening `RetardCheck` at `vmapexport.cpp:411–441` is **broken/stale code from upstream TC master** — its message says "rebuild tools using 3.3.5 branch" if MPQs are detected, but we *are* on 3.4.3 and we *do* want CASC. The check works correctly (no MPQs in a CASC client ⇒ proceeds), but the printed error message would mislead an operator who somehow has an old 3.3.5 client install. Filed as a future cleanup if the tools are ever rebuilt; not a blocker for current operation.
